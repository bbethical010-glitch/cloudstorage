You are a senior distributed systems engineer specializing in streaming
protocols, WebRTC data channels, and backend stream processing. You
design fault-tolerant, production-grade data pipelines.

Fix a critical bug where folder uploads complete at 100% but only
directory structures are created and files are missing.

The UI also gets stuck in a loading state after upload.

System: - Frontend: React (streaming TAR upload) - Transport: - WebRTC
DataChannel (primary) - WebSocket relay fallback - Backend: Android Ktor
server

Observed behavior: - Upload reaches 100% - DataChannel closes - Only
folders appear on backend - Files are missing - UI never exits loading
state

Console logs show: - DataChannel closed before completion - No explicit
upload completion signal

The system incorrectly relies on connection closure as a signal for
upload completion.

There is: - no explicit END-OF-STREAM signal - no backend
acknowledgment - no guarantee that all chunks are received before
extraction completes

This causes partial extraction and data loss.

Implement a robust streaming protocol with explicit lifecycle control.

STEP 1 — Define Protocol Messages

Define message types:

-   START_UPLOAD
-   DATA_CHUNK
-   END_UPLOAD
-   ACK_COMPLETE
-   ERROR

Each message must include: - uploadId - sequenceNumber (for ordering) -
optional payload (binary for DATA_CHUNK)

------------------------------------------------------------------------

STEP 2 — Frontend Sender

-   Generate unique uploadId
-   Send START_UPLOAD
-   Stream TAR as DATA_CHUNK messages
-   Track total bytes sent

IMPORTANT: - Do NOT close DataChannel after sending last chunk - Wait
for ACK_COMPLETE from backend

After last chunk: - send END_UPLOAD explicitly

------------------------------------------------------------------------

STEP 3 — Backend Receiver (Ktor)

-   On START_UPLOAD:
    -   initialize stream context (file, buffer, state)
-   On DATA_CHUNK:
    -   append to stream
    -   maintain sequence order
-   On END_UPLOAD:
    -   finalize stream
    -   close file
    -   trigger TAR extraction
    -   verify integrity

ONLY after successful extraction: - send ACK_COMPLETE

------------------------------------------------------------------------

STEP 4 — Extraction Safety

-   Do NOT extract while stream is incomplete
-   Ensure TAR footer is received
-   Validate archive consistency

------------------------------------------------------------------------

STEP 5 — UI Fix

-   Replace “upload complete” condition:

FROM: uploadProgress === 100%

TO: received ACK_COMPLETE from backend

------------------------------------------------------------------------

STEP 6 — Handle DataChannel Closure

-   If channel closes BEFORE END_UPLOAD:
    -   mark upload as failed
    -   cleanup partial files
    -   notify UI

------------------------------------------------------------------------

STEP 7 — Add Timeout + Retry

-   If no ACK after END_UPLOAD:
    -   retry or fail gracefully

------------------------------------------------------------------------

STEP 8 — Logging

Add logs for: - START_UPLOAD received - last DATA_CHUNK received -
END_UPLOAD received - extraction complete - ACK sent

 - Must work for large streams (GB-scale) - Must tolerate network
instability - Must NOT rely on connection close as signal

Return:

1.  Frontend implementation (stream sender)
2.  Backend implementation (stream receiver)
3.  Protocol definition
4.  UI completion logic update

All code must be production-ready and complete.

The solution is correct if: - All files are correctly extracted - Upload
does not depend on connection closing - UI exits loading state
correctly - No partial folder-only uploads occur

 - Do not output pseudo-code - Do not skip edge cases - Focus on
correctness over brevity
